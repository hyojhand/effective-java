# 아이템 4. 인스턴스화를 막으려거든 private 생성자를 사용하라

### 정적 메서드만 남은 유틸리티 클래스는 인스턴스로 만들어 쓰려고 설계한 클래스가 아니다.

유틸리티 성격의 클래스는 static한 메서드만 가지고 있다.

클래스로 바로 호출하여 사용하면 되기 때문에, 인스턴스를 만들어 호출하는 것은 불필요한 과정이다.  
오히려 인스턴스 메서드인지 static 메서드인지 헷갈릴 수 있기 때문에 인스턴스화를 방지하는 것이 좋다.

### 추상 클래스로 만드는 것으로는 인스턴스화를 막을 수 없다.

하위 클래스를 만들어 인스턴스화하면 그만이다.

**추상 클래스를 상속한 자식 클래스의 인스턴스를 만들때, 상위 클래스의 생성자를 기본으로 호출**하기 때문에,  
추상 클래스로 만들어도 상속해서 자식클래스로 인스턴스화 할 수 있다.

오히려 사용자가 abstract 키워드로 인해 상속해서 쓰라는 뜻으로 오해할 수 있는 것이 더 큰 문제이다.

### private 생성자를 추가하면 클래스의 인스턴스화를 막을 수 있다.

**부모 생성자가 private이기 때문에 상속 자체가 불가능해진다! (상속을 방지할 때도 같은 방법을 사용할 수 있다)**

클래스 안에서는 생성자를 접근할 수 있지만 이것도 막고싶다면, 에러(AssertionError)를 던져서 실수로라도 이러한 상황을 만나지 않도록 하는 방법이 있다.

하지만, 생성자가 분명 존재하는데 호출할 수 없다는 것이 그다지 직관적이지 않을 수 있다!  
따라서, 생성자에 주석으로 인스턴스화 불가한 이유를 설명하는 것이 좋다.
